# Аутентификация

На этой странице объясняется, как на самом деле происходит вход в Axelix Master: какие существуют варианты
аутентификации, как устанавливается и проверяется сеанс работы в браузере при каждом запросе, как работает модель ролей
и полномочий, ограничивающая доступ к пользовательскому интерфейсу, и как сервер MCP выполняет аутентификацию агентов
ИИ. Сами ключи свойств находятся в разделе [Настройка
Master](../../setting-up-master-ui/configuring-master/configuring-master.mdx#справочник-по-конфигурации) — здесь мы рассказываем о ментальной
модели.

## Три варианта входа

Master поддерживает три независимых поставщика аутентификации. Можно одновременно использовать любую комбинацию. На
экране входа в систему `/login` отображаются элементы управления для всех активных поставщиков.

<div className={signInOptionsStyles.signInOptionsTable}>

| Вариант                | По умолчанию   | Переключить свойство                                                                      | Используйте его, когда                                                                                                |
|------------------------|----------------|-------------------------------------------------------------------------------------------|-----------------------------------------------------------------------------------------------------------------------|
| Суперадминистратор     | Всегда активен | (контролируется `axelix.master.auth.options.super-admin.credentials.{username,password}`) | Первоначальная настройка, резервная учётная запись, развёртывание с помощью одного оператора.                         |
| Локальные пользователи | ВЫКЛ           | `axelix.master.auth.options.local.enabled=true`                                           | Master владеет хранилищем паролей.                                                                                    |
| OAuth2 / OIDC          | ВЫКЛ           | `axelix.master.auth.options.oauth2.enabled=true`                                          | Production. Единый вход через Keycloak, Auth0, Okta, Google или любой другой провайдер, поддерживающий протокол OIDC. |

Если включено несколько параметров, они отображаются на экране в виде списка: форма для ввода имени пользователя и
пароля для суперадминистраторов и локальных пользователей, а затем одна кнопка **«Войти с помощью OAuth2/OIDC
Provider»** при включённом OIDC.

### Суперадминистратор

Встроенная учётная запись суперадминистратора — единственная, которая всегда активна. Master сравнивает введённые
учётные данные напрямую с `axelix.master.auth.options.super-admin.credentials.username` / `.password` — без
использования базы данных. При успешном входе в систему пользователю предоставляется роль `SUPER_ADMIN`,
которая даёт все полномочия, определённые в Master (см. [Роли и полномочия](#roles-and-authorities)).

Эта учётная запись предназначена для начальной настройки Master и восстановления доступа в случае неправильной настройки
других провайдеров. Прежде чем предоставлять доступ к Master кому бы то ни было, измените настройки по умолчанию —
`admin / admin` даёт полный контроль над каждым мониторируемым сервисом.

Пароль может быть указан в одном из следующих форматов:

* **Plaintext** — пароль указывается в исходном виде.
* **Encoded** — перед значением указывается идентификатор в фигурных скобках, определяющий способ его проверки: `{noop}`
для пароля в открытом виде, `{bcrypt}` для хеша BCrypt.

Если префикс отсутствует, значение рассматривается как plaintext.
В целях безопасности настоятельно рекомендуется использовать хешированный формат пароля (например, `{bcrypt}`) вместо
хранения пароля в открытом виде.

### Локальные пользователи

Локальные пользователи хранятся в собственной базе данных Master (`UserEntity`). Пароли хешируются с помощью алгоритма
BCrypt. Когда `axelix.master.auth.options.local.enabled=true`, `DatabaseUserAuthenticator` ищет введённое имя
пользователя, BCrypt проверяет пароль и сопоставляет сохранённую строку роли (`admin`, `editor`, `viewer`) с одной из
встроенных ролей. При входе в систему с неизвестной строкой роли возникает `UserRoleNotFoundException`.

Управление локальными пользователями осуществляется через экран «Пользователи» в пользовательском интерфейсе. Для их
создания, обновления и удаления требуется полномочие `USERS_MANAGEMENT` — по умолчанию оно есть только у `SUPER_ADMIN`.

### OAuth2 / OIDC

Master реализует стандартный поток авторизации с использованием кода. Включите его с помощью
`axelix.master.auth.options.oauth2.enabled=true` и укажите `issuer-uri`, `client-id`, `client-secret`, а также
общедоступный `base-url` этого развёртывания Master.

Полный процесс:

1. Пользователь нажимает **«Войти с помощью OAuth2/OIDC Provider»** на `/login`. Браузер перенаправляется на конечную
   точку авторизации поставщика удостоверений с параметрами `client-id` и `redirect_uri`, установленными на
   `<base-url>/api/external/oauth2/callback`.
2. Поставщик удостоверений аутентифицирует пользователя и перенаправляет его обратно на страницу обратного вызова с
   `code`.
3. `OAuth2CallbackController` обменивает `code` на токен ID и токен доступа через конечную точку токенов поставщика
   удостоверений, проверяет токен ID и извлекает имя пользователя.
4. Роль определяется путём сопоставления выражения JMESPath в `axelix.master.auth.options.oauth2.role-attribute-path` с
   ответом userinfo. Результат сопоставляется с `admin`, `editor` или `viewer`. Если `role-attribute-path` не задано,
   каждый пользователь OIDC получает роль `VIEWER`.
5. Перед выдачей cookie сеанса Master создаёт или обновляет пользователя в своей базе данных с origin `OIDC`,
   обновляет сохранённый email из userinfo, если он есть, и фиксирует `Last login`.
6. Master создаёт собственный JWT-токен сеанса (с тем же кодировщиком, что и в потоках паролей) и перенаправляет браузер
   на `/wallboard` с установленными файлами cookie сеанса.

Процесс авторизации с использованием кода авторизации запускается только через браузер. Сами токены OIDC никогда не
сохраняются — Master использует их один раз для идентификации пользователя, а затем выдаёт собственный JWT-токен сеанса.

Поэтому каждый успешный вход через браузер оставляет после себя учётную запись `OIDC` на экране
**Пользователи**. Если то же имя пользователя уже принадлежит не-OIDC учётной записи, Master отклоняет вход вместо
объединения идентичностей.

При использовании `axelix.master.auth.options.oauth2.enabled=true` необходимо указать следующие свойства: `issuer-uri`,
`client-id`, `client-secret`, `base-url`. `scopes` по умолчанию имеет значение `openid` и требует переопределения только
в том случае, если вашему провайдеру идентификации требуются дополнительные области доступа. `role-attribute-path` —
необязательный параметр. Если он не указан, каждый вошедший в систему пользователь будет иметь роль `VIEWER`.

## Сеанс: cookie и JWT

Все пути входа в систему заканчиваются одинаково: Master генерирует JWT и возвращает его в виде cookie. Этот же
cookie-файл используется для каждого последующего запроса к пользовательскому интерфейсу.

### При входе в систему

`POST /api/external/users/login` (и при обратном вызове OIDC) создают два файла cookie через `Set-Cookie`:

- **`auth_token`** — сам JWT. `HttpOnly`, `SameSite=Strict`, `Path=/`, `MaxAge=axelix.master.auth.jwt.lifespan` (по
  умолчанию `12h`). `Secure` контролируется свойством `axelix.master.auth.cookie.secure` — оставляйте его `false` только
  для обычного HTTP-доступа к localhost.
- **`authorities`** — закодированный в Base64 JSON-массив с именами полномочий, которыми обладает вошедший в систему
  пользователь, например `["ENV_VALUES_READ","SCHEDULED_TASKS_MODIFY"]`. *Не* `HttpOnly`, потому что интерфейс считывает
  его, чтобы знать, какие кнопки отображать. Это всего лишь подсказка для пользовательского интерфейса. Серверная часть
  при каждом запросе заново определяет полномочия на основе JWT, поэтому изменение этого файла cookie влияет только на
  то, что предлагает пользовательский интерфейс, а не на то, что позволяет сервер.

Сам JWT подписан с помощью алгоритма HMAC (`HMAC256`, `HMAC384` или `HMAC512`, задаётся через
`axelix.master.auth.jwt.algorithm`) с использованием секрета в `axelix.master.auth.jwt.signing-key`. Его поля — это имя
пользователя (`sub`), метки времени выдачи и истечения срока действия, а также поле `roles`, в котором кодируется каждая
предоставленная роль вместе с её полномочиями и вложенными компонентами.

### Через что проходит каждый запрос

`CookieBasedJwtAuthorizationFilter` выполняется при каждом `/api/...` запросе, кроме небольшого списка разрешённых (сама
конечная точка входа, обратный вызов OAuth2, конечная точка общедоступных настроек, конечная точка самостоятельной
регистрации и `/api/mcp/**` — для них предусмотрены собственные фильтры или они намеренно анонимны). Для всего
остального:

1. Фильтр считывает файл cookie `auth_token`. Отсутствует или пуст → `JwtProcessingException`, отображается как `401
   Unauthorized`.
2. `DefaultJwtDecoderService` проверяет подпись и срок действия по `axelix.master.auth.jwt.signing-key`. Подделанный,
   неподписанный или просроченный токен → `401`.
3. `MasterAuthorityResolver` ищет путь запроса + метод HTTP во встроенной таблице правил «для этой конечной точки
   требуется это полномочие». Для конечных точек, которых нет в таблице, требуется только действительный токен.
4. `DefaultAuthorizer` объединяет роли пользователя в полный набор полномочий и проверяет, включает ли он необходимые.
   Не хватает одного → `AuthorizationException`, отображается как `403 Forbidden`.
5. В случае успешного выполнения запрос выполняется внутри `DefaultSecurityContext` с аутентифицированным `User` и
   исходным токеном, поэтому нижестоящий код может считать данные вызывающей стороны без повторного анализа.

### Выход и ротация

`POST /api/external/users/logout` перезаписывает оба файла cookie пустыми значениями и `MaxAge=0`, поэтому браузер сразу
же удаляет их. Сам JWT технически остаётся действительным до истечения его поля `exp` — Master не ведёт серверный
блок-лист.

Чтобы принудительно завершить все активные сеансы (после компрометации ключа, устаревания общего секретного ключа или
ежеквартальной ротации), измените `axelix.master.auth.jwt.signing-key`. Каждый выданный токен сразу же не пройдёт
проверку при следующем запросе, поскольку подпись перестанет соответствовать новому секретному ключу. Пользователи
вернутся на `/login`.

## Роли и полномочия

Master использует фиксированный набор ролей и полномочий. Роли представляют собой набор полномочий; проверка авторизации
всегда выполняется на основе полномочий, а не названий ролей. Это означает, что пользовательская роль, подключённая
через сопоставление OIDC или уровень базы данных, должна соответствовать одному из встроенных названий — на сегодняшний
день нет возможности определить новое полномочие в конфигурации.

<div className={rolesStyles.rolesTable}>

| Роль          | Предоставленные полномочия                                                                                                     | Кто это получит                                                                                                                      |
|---------------|--------------------------------------------------------------------------------------------------------------------------------|--------------------------------------------------------------------------------------------------------------------------------------|
| `SUPER_ADMIN` | Все полномочия, определённые в Master, включая `USERS_VIEW` и `USERS_MANAGEMENT`.                                              | Встроенная учётная запись суперадминистратора.                                                                                       |
| `ADMIN`       | `ENV_VALUES_READ`, `CONFIG_PROPS_VALUES_READ`, `SCHEDULED_TASKS_MODIFY`, `CACHES_CLEAR`, `CACHES_TOGGLE`, `GARBAGE_COLLECTOR`. | Пользователи локальной базы данных с ролью `admin`. Пользователи OIDC, для которых `role-attribute-path` соответствует `admin`.      |
| `EDITOR`      | `SCHEDULED_TASKS_MODIFY`, `CACHES_CLEAR`, `CACHES_TOGGLE`, `GARBAGE_COLLECTOR`.                                                | Пользователи локальной базы данных с ролью `editor`. Пользователи OIDC, для которых `role-attribute-path` соответствует `editor`.    |
| `VIEWER`      | Нет — доступ только для чтения ко всему, что не защищено полномочием.                                                          | Пользователи локальной базы данных с ролью `viewer`. Пользователи OIDC без `role-attribute-path` или с несопоставленными значениями. |

</div>

Что защищает каждое полномочие:

- **`ENV_VALUES_READ`** — отображение значений свойств на экране «Окружение».
- **`CONFIG_PROPS_VALUES_READ`** — отображение значений свойств на экране «Свойства конфигурации».
- **`SCHEDULED_TASKS_MODIFY`** — включение, отключение, принудительное выполнение и изменение расписания запланированных
  задач.
- **`CACHES_CLEAR`** — очистка отдельных кэшей, кэш менеджеров или всего кэша сервиса.
- **`CACHES_TOGGLE`** — включение и отключение отдельных кэшей и кэш менеджеров во время выполнения.
- **`GARBAGE_COLLECTOR`** — запуск сборки мусора, включение и отключение мониторинга журнала сборки мусора.
- **`USERS_VIEW`** — чтение ленты пользователей.
- **`USERS_MANAGEMENT`** — создание, обновление и удаление локальных пользователей.

`USERS_VIEW` и `USERS_MANAGEMENT` сегодня зарезервированы для `SUPER_ADMIN` — `ADMIN` не может управлять пользователями
без повышения уровня доступа.

## Аутентификация на MCP-сервере

Встроенный MCP-сервер по адресу `/api/mcp` защищён собственным фильтром `McpAuthorizationFilter`, поскольку учётные
данные, передаваемые MCP-клиентами, не являются cookie-файлами сеанса браузера. Фильтр анализирует заголовок
`Authorization` и перенаправляет запрос обработчику, соответствующему схеме.

- **`Basic`** — `BasicMcpAuthenticationHandler` декодирует `Authorization: Basic <base64(user:pass)>` и делегирует тому
  же `CompositeUserAuthenticator`, который использует пользовательский интерфейс. Работают как суперадминистраторы, так
  и локальные учётные записи, Сопоставление ролей идентично входу через браузер.
- **`Bearer`** — `BearerMcpAuthenticationHandler` обрабатывает учётные данные как токен доступа OIDC. Он прогоняет токен
  через `JmesPathOidcRoleExtractor` (по тому же пути, что и при входе через браузер), чтобы определить роль, а затем
  выдаёт синтетический `PasswordlessUser` с именем `MCP_AGENT` с этой ролью. Доступно только когда
  `axelix.master.auth.options.oauth2.enabled=true`.

После идентификации пользователя фильтр выполняет ту же проверку полномочий, что и веб-фильтр — требования к полномочиям
по каждому инструменту разрешаются через `McpEndpointResolver` + `McpEndpointAuthorityResolver`. Инструмент, на
использование которого у вызывающей стороны нет прав, возвращает `403 Forbidden`. При использовании неизвестных или
недействительных учётных данных возвращается `401 Unauthorized`.

Если OIDC включён и запрос поступает без заголовка `Authorization`, ответ `401` также содержит:

```
WWW-Authenticate: Bearer resource_metadata="<base-url>/api/external/mcp-oauth2/.well-known/oauth-protected-resource"
```

MCP-клиенты, поддерживающие RFC 9728, используют этот заголовок, чтобы получить доступ к конфигурации OAuth2 в Master и
самостоятельно провести пользователя через authorization flow.
